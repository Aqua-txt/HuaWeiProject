## 2. 背景与动机

### 市场背景

- 高校学生每学期会产生大量闲置物品：教材教辅（使用一学期后基本闲置）、数码产品（毕业换代）、生活用品（搬家不便带走）。这些物品对原主人价值低，但对有需求的同学价值高。
- 当前校园二手交易主要依赖微信群/QQ群/表白墙发布信息，存在**信息分散、查找困难、交易撮合效率低**的问题。
- 校外平台（闲鱼、转转）虽然成熟，但缺乏校园属性——无法验证校友身份、交易需邮寄、缺少校内当面交易的信任场景。

### 数据支撑

- 据教育部 2025 年数据，全国高校在校生超过 4500 万人，人均闲置物品流通需求旺盛。
- 校内调研（假设数据）：约 70% 的学生有二手交易需求，但仅 20% 通过现有渠道成功完成过交易。

### 战略意义

- 为校园生活服务生态补全「闲置物品流通」这一高频刚需场景。
- 以二手交易为切入点，积累学生信任关系链，未来可扩展至校园租房、兼职、拼单等场景。

## 3. 问题定义

### 核心问题

在校学生有闲置物品想卖掉、有二手物品想买入，但缺乏一个**可信、便捷、校园专属**的交易撮合平台。

### 痛点分解

| 角色 | 痛点 | 场景 |
|------|------|------|
| 卖方 | 发布渠道分散，需在多个群重复发消息；消息很快被刷走 | 毕业季搬宿舍，旧书/电器无处处理 |
| 卖方 | 难以触达真正有需求的同学 | 一本专业书只有本专业同学需要 |
| 买方 | 想淘便宜教材/用品，不知道哪里有 | 新学期买教材，全新太贵 |
| 买方 | 无法搜索，只能爬楼翻聊天记录 | 想找特定型号的二手笔记本 |
| 双方 | 交易沟通不便，需加微信私聊 | 看中物品想问问成色/价格 |
| 双方 | 没有信任背书，担心遇到校外骗子 | 首次交易时对对方身份存疑 |

### 用户价值主张

**"让校园闲置物品像发朋友圈一样简单发布，像逛淘宝一样方便查找，像聊微信一样直接沟通。"**

## 4. 目标与指标

### 产品目标

在校园内建立起一个可信、高效的二手交易信息撮合平台，降低交易信息门槛，提升校内闲置物品流通率。

### 成功指标

| 指标类型 | 指标名称 | 目标值（上线后 3 个月） | 说明 |
|----------|----------|------------------------|------|
| **北极星指标** | 月活跃交易用户数 (MAU) | ≥ 在校生数的 15% | 衡量平台核心价值是否被用户认可 |
| **过程指标** | 商品发布数 | ≥ 500 件/月 | 供给侧活跃度 |
| **过程指标** | 搜索使用率 | ≥ 发布数的 3 倍 | 需求侧活跃度 |
| **过程指标** | 聊天转化率（咨询→面议） | ≥ 40% | 沟通效率 |
| **过程指标** | 商品浏览→咨询转化率 | ≥ 15% | 商品信息质量 + 匹配效率 |
| **反向指标** | 用户举报率 | ≤ 0.5% | 防止虚假/违规商品污染平台 |
| **反向指标** | 聊天消息平均响应时长 | ≤ 4 小时 | 沟通体验的底线保障 |

## 5. 目标用户

### 用户画像

| 角色 | 卖方（学生 A） | 买方（学生 B） |
|------|---------------|---------------|
| 身份 | 大三/大四学生 | 大一/大二学生 |
| 动机 | 毕业清宿舍 / 更新装备，想快速处理闲置 | 省钱购买教材 / 日用品 |
| 频率 | 每学期 1-3 次（集中发布） | 每月 2-5 次（按需搜索） |
| 技术能力 | 普通互联网用户 | 普通互联网用户 |
| 核心诉求 | 快速拍照上架，有人来问即可 | 高效找到想要的物品，联系卖家 |

### 使用场景

1. **学期末/毕业季**：批量发布闲置物品（典型高频场景）
2. **开学季**：搜索购买二手教材（典型高频场景）
3. **日常**：偶尔发布/搜索电子产品、生活用品
4. **即时**：看到感兴趣的商品，立即发起聊天咨询

### 当前替代方案

- 微信群/QQ群接龙 → 信息混乱、不可搜索
- 校园表白墙投稿 → 发布麻烦、无法交互
- 闲鱼/转转 → 需邮寄、无校园身份验证
- 跳蚤市场线下摆摊 → 时间空间受限

## 6. 需求范围

### 包含（In Scope）— v1.0 MVP

#### 核心功能

- [ ] **用户注册与登录**
  - 支持学号 + 密码注册
  - 学校邮箱验证（可选）
  - 个人资料管理（头像、昵称、联系方式）

- [ ] **商品发布**
  - 发布商品表单：标题、描述、价格、成色（全新/几乎全新/轻微使用痕迹/明显使用痕迹）、交易方式（面议/自提）
  - 支持上传 1-6 张商品图片
  - 商品分类（教材/数码/生活/其他）
  - 发布后进入待上架状态，默认上架
  - 支持编辑、下架、删除自己发布的商品

- [ ] **商品浏览与搜索**
  - 首页信息流：按最新发布排序的商品列表
  - 分类筛选：按商品分类快速筛选
  - 关键词搜索：按标题/描述模糊搜索
  - 搜索结果支持按价格、时间排序
  - 商品详情页：大图展示、完整信息、卖家信息

- [ ] **即时聊天**
  - 从商品详情页直接发起聊天
  - 一对一文本消息聊天
  - 聊天列表页：管理所有对话
  - 消息通知（未读消息提醒）

- [ ] **个人中心**
  - 我发布的商品管理
  - 我的收藏
  - 个人资料编辑

### 不包含（Out of Scope）— 后续版本

- 在线支付与交易担保（v2.0 考虑接入校园卡/微信支付）
- 评价与信用体系（v2.0）
- 学校管理员后台（v2.0）
- 拼团/求购功能（v2.0+）
- 消息推送/站外通知（v2.0+）
- 定位与地图功能（v2.0+）

### 依赖

- 学校学生数据接口（用于学号验证，可选依赖）
- 图片文件存储方案（本地文件系统或对象存储）
- WebSocket 服务或轮询机制（用于聊天实时性）

## 7. 方案设计

### 7.1 整体架构

```
[浏览器] Vue3 SPA
    ↕ REST API + WebSocket
[Flask 应用] (API 服务)
    ↕
[SQLite] 数据库
[本地文件系统] 图片存储
```

### 7.2 信息架构

```
首页（商品信息流）
├── 搜索栏（关键词 + 分类筛选）
├── 分类导航（教材 / 数码 / 生活 / 其他）
└── 商品卡片列表（图片、标题、价格、发布时间）

商品详情页
├── 商品大图（轮播）
├── 商品信息（标题、描述、价格、成色、分类、发布时间）
├── 卖家信息（头像、昵称、联系按钮）
└── 操作按钮（联系卖家 / 收藏）

发布商品页
├── 图片上传（1-6 张）
├── 标题输入
├── 描述输入
├── 价格输入
├── 成色选择
├── 分类选择
└── 发布按钮

聊天页
├── 聊天列表（按最后消息时间排序）
└── 聊天详情（消息记录 + 输入框）

个人中心
├── 我的发布（管理自己的商品）
├── 我的收藏
└── 个人资料编辑
```

### 7.3 核心业务流程

#### 商品发布流程

```
用户点击"发布"
  → 填写商品信息 + 上传图片
  → 点击"发布"
  → 后端创建商品记录（状态：上架）
  → 跳转至商品详情页
```

#### 搜索流程

```
用户在首页搜索框输入关键词
  → 前端发送搜索请求（关键词 + 筛选条件）
  → 后端 SQLite 模糊查询（LIKE 匹配标题 + 描述）
  → 返回匹配的商品列表
  → 前端渲染搜索结果
  → 用户可进一步按价格/时间排序
```

#### 聊天流程

```
用户在商品详情页点击"联系卖家"
  → 创建/获取会话（会话 ID = 当前用户 + 卖家）
  → 跳转至聊天页
  → 双方发送消息
  → 后端存储消息 + 轮询/WebSocket 推送
  → 接收方查看消息（未读→已读）
```

### 7.4 数据库核心表设计

```
用户表 (user):
  id, student_id, nickname, avatar, password_hash, 
  phone, email, created_at

商品表 (product):
  id, seller_id (FK), title, description, price, 
  category, condition, images (JSON), status,
  created_at, updated_at

聊天会话表 (conversation):
  id, product_id (FK), buyer_id (FK), seller_id (FK),
  created_at, updated_at

聊天消息表 (message):
  id, conversation_id (FK), sender_id (FK), 
  content, msg_type (text/image), created_at, is_read

收藏表 (favorite):
  id, user_id (FK), product_id (FK), created_at
```

### 7.5 异常场景处理

| 场景 | 处理方式 |
|------|----------|
| 用户上传非图片文件 | 前端校验文件类型 + 后端二次校验，返回具体错误提示 |
| 图片过大 | 限制单张 ≤ 5MB，前端压缩后上传 |
| 搜索无结果 | 显示"未找到相关商品"空状态，推荐热门分类 |
| 对方已下架商品 | 进入商品详情提示"商品已下架"，聊天入口灰化 |
| 网络异常 | 请求失败时 Toast 提示"网络异常，请稍后重试" |
| 发布空表单 | 前端必填校验，高亮未填字段 |
| 非法价格输入（负数/超大） | 前端限制 + 后端校验，范围 0.01-99999.99 |

## 8. 技术要求

### 系统模块

| 模块 | 技术 | 说明 |
|------|------|------|
| 前端 SPA | Vue3 + Vue Router + Pinia | 单页应用，组件化开发 |
| UI 框架 | Element Plus | 桌面端优先，后台管理风格，兼容常用分辨率 |
| 后端 API | Flask + Flask-RESTful | RESTful API 设计 |
| 数据库 | SQLite + Flask-SQLAlchemy | 轻量级，无需额外部署数据库服务 |
| 聊天实时性 | Flask-SocketIO (WebSocket) 或 短轮询 | 首选 WebSocket，退化方案为 3s 轮询 |
| 图片存储 | 本地文件系统 + 图片 URL | 后续可迁移至对象存储 |
| 认证 | Flask-Login + JWT | 基于 Token 的认证方案 |

### 工作量预估

| 模块 | 预估人日 | 说明 |
|------|---------|------|
| 用户系统（注册/登录/个人中心） | 3-4 人日 | 前后端 + 数据库 |
| 商品 CRUD（发布/编辑/删除/列表） | 4-5 人日 | 含图片上传 |
| 搜索与筛选 | 1-2 人日 | SQLite 模糊查询 |
| 聊天系统 | 4-6 人日 | WebSocket + 消息存储 |
| 基础前端框架搭建 | 2-3 人日 | 路由、状态管理、布局 |
| 联调与测试 | 2-3 人日 | |
| **总计** | **16-23 人日** | 约 3-4 周（1 人全栈） |

### 性能要求

- 首屏加载时间 ≤ 2s（开发环境）
- 商品列表 API 响应 ≤ 200ms（1000 条数据以内）
- 搜索 API 响应 ≤ 500ms
- 聊天消息延迟 ≤ 1s（WebSocket 模式）
- 并发量预估：初期 ≤ 100 QPS（校内使用）

## 9. 验收标准

### 用户注册与登录

- [ ] Given 未注册用户，When 填写学号、昵称、密码并提交，Then 注册成功并自动登录
- [ ] Given 已注册用户，When 输入正确学号密码，Then 登录成功跳转首页
- [ ] Given 已注册用户，When 输入错误密码，Then 提示"学号或密码错误"
- [ ] Given 未登录用户，When 访问需要登录的页面（发布、聊天、个人中心），Then 重定向至登录页

### 商品发布

- [ ] Given 已登录用户，When 填写完整商品信息并提交，Then 商品上架成功，跳转至详情页
- [ ] Given 已登录用户，When 未上传图片就提交，Then 提示"请至少上传一张图片"
- [ ] Given 已登录用户，When 价格输入非数字，Then 前端拦截并提示"请输入有效价格"
- [ ] Given 商品所有者，When 点击下架按钮，Then 商品状态变为"已下架"，不再出现在搜索列表中
- [ ] Given 商品所有者，When 编辑商品信息并保存，Then 商品信息更新

### 搜索与浏览

- [ ] Given 首页，When 用户输入关键词搜索，Then 返回按标题/描述模糊匹配的商品列表
- [ ] Given 搜索结果页，When 用户切换分类筛选，Then 结果按分类过滤
- [ ] Given 搜索结果页，When 用户选择"价格从低到高"，Then 结果按价格升序排列
- [ ] Given 搜索无结果，When 展示页面，Then 显示空状态提示和推荐分类
- [ ] Given 商品已下架或删除，When 访问该商品详情页，Then 提示"商品已下架"

### 聊天

- [ ] Given 已登录用户在商品详情页，When 点击"联系卖家"，Then 进入与该卖家的聊天页面
- [ ] Given 买方进入聊天页，When 输入消息并发送，Then 消息显示在对话中，卖家可收到
- [ ] Given 卖家收到消息，When 回复消息，Then 买方可见回复内容
- [ ] Given 用户有未读消息，When 查看聊天列表，Then 该对话显示未读标记
- [ ] Given 用户点开对话，When 查看消息，Then 未读标记消除

### 边界与异常

- [ ] Given 用户上传超过 6 张图片，When 选择第 7 张，Then 提示"最多上传 6 张图片"
- [ ] Given 用户上传超过 5MB 的图片，When 上传，Then 提示"图片大小不能超过 5MB"
- [ ] Given 网络断开时用户发送聊天消息，When 点击发送，Then 提示"消息发送失败"并保留输入
- [ ] Given 数据库无数据，When 访问首页，Then 显示"还没有商品，快来发布第一个吧"

## 10. 附录

### 竞品分析摘要

| 竞品 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|------------|
| 闲鱼 | 用户量大，交易体系成熟 | 非校园专属，交易需邮寄，有平台抽成 | 纯校园环境，当面交易，0 费用 |
| 微信群/QQ群 | 用户就在群里，零门槛 | 信息杂乱，不可搜索，容易被刷屏 | 结构化信息，可搜索，持久化 |
| 校内跳蚤市场 | 面对面交易，信任度高 | 时间空间受限，频次低 | 线上 24h，不受时空限制 |

### 技术选型理由

- **Vue3**：相比 Vue2 性能提升、Composition API 更利于逻辑复用、社区生态成熟
- **Flask**：轻量级 Python Web 框架，适合中小型项目，开发效率高，与 SQLite 配合极佳
- **SQLite**：零配置、无需单独部署数据库服务、完全满足校内规模的数据量（万级），适合 MVP 阶段
- **为什么不用 MySQL/PostgreSQL**：MVP 阶段 SQLite 足够，且减少了部署运维成本。后续用户量增长后可平滑迁移至 MySQL

