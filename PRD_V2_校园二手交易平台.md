## 2. 背景与动机

### V1.0 回顾

V1.0 已上线运行并验证了核心假设：校园内存在旺盛的二手交易需求，学生愿意使用结构化平台发布和搜索商品。V1.0 实现了信息撮合的最小闭环——发布、浏览、搜索、聊天——解决了"信息分散、不可搜索"的核心痛点。

### V1.0 的局限

V1.0 本质上是一个**信息撮合工具**，交易撮合的最后一公里仍然依赖线下：

- **交易无保障**：买卖双方谈拢后完全在线下完成交易，遇到货物不符、一方反悔、价格纠纷等情况平台无能为力
- **无信用积累**：每次交易都是"一次性博弈"，好卖家无法积累信誉、差卖家没有代价，用户之间缺乏信任信号
- **无运营抓手**：管理员无法监控平台健康度，违规内容只能等用户反馈，数据盲飞
- **供给侧单一**：只有卖家驱动的内容供给，缺少买家驱动的需求表达（求购）

### V2.0 的战略目标

从**信息撮合工具**升级为**交易服务平台**，让用户不仅能"找到货"，还能"放心交易"。以交易闭环积累信用数据，以信用数据反哺交易效率，形成正向飞轮。

| 从 V1.0 | 到 V2.0 |
|---------|---------|
| 信息撮合 | 交易闭环 |
| 一次性博弈 | 信用积累 |
| 买家找卖家 | 双向匹配（求购 + 商品） |
| 无管理能力 | 运营后台 + 数据看板 |

## 3. 问题定义

### 核心问题

V1.0 解决了"信息撮合"问题，但交易双方在**资金安全、信任判断、纠纷处理**三方面仍然缺乏平台级保障，导致交易转化率低、用户粘性弱、平台价值停留在浅层。

### 痛点分解

| 角色 | 痛点 | 场景 |
|------|------|------|
| 买方 | 付款后担心卖家不发货/货不对板 | 线上谈好价格付了款，见面发现商品与描述不符 |
| 买方 | 不知道卖家是否靠谱 | 想交易但看到卖家没有过往记录，不敢下手 |
| 买方 | 想买的特定物品没人发布 | 需要一本冷门教材，翻遍列表找不到 |
| 卖方 | 定价困难，不知道买家愿意出多少钱 | 二手笔记本不知道该卖 500 还是 800 |
| 卖方 | 优质卖家无法展示自己的信誉积累 | 已成交 10 单零纠纷，但新买家看不到 |
| 管理员 | 无法感知平台健康度 | 不知道多少商品违规、多少交易产生纠纷 |
| 管理员 | 处理违规效率低 | 用户 QQ 群里投诉，缺乏结构化处理流程 |

### 用户价值主张

**"让每一笔校园交易都有保障、有记录、有信用——像当面交易一样安全，像线上平台一样高效。"**

## 4. 目标与指标

### 产品目标

在 V1.0 信息撮合基础上，建立"支付担保 → 交易完成 → 信用评价"的完整闭环，同时引入求购专区实现双向匹配，配合运营后台让平台可管理、可度量。

### 成功指标

| 指标类型 | 指标名称 | 目标值（V2 上线后 3 个月） | 说明 |
|----------|----------|------------------------|------|
| **北极星指标** | 月担保交易笔数 | ≥ 200 笔/月 | 衡量交易闭环是否被用户接受 |
| **北极星指标** | 交易完成率 | ≥ 70%（付款→完成） | 衡量交易流程是否顺畅 |
| **过程指标** | 微信支付发起率 | ≥ 商品咨询量的 20% | 支付入口的转化效率 |
| **过程指标** | 评价覆盖率 | ≥ 完成交易的 60% | 评价体系是否被用户使用 |
| **过程指标** | 求购发布数 | ≥ 100 条/月 | 求购专区的活跃度 |
| **过程指标** | 求购匹配率 | ≥ 30%（有人响应） | 求购需求的满足效率 |
| **反向指标** | 交易纠纷率 | ≤ 5%（申诉/退款笔数） | 交易质量底线 |
| **反向指标** | 违规商品存活时间 | ≤ 24 小时（从发布的举报） | 先发后审的风险窗口 |
| **管理指标** | 举报处理时效 | ≤ 12 小时（工作日） | 运营响应速度 |

## 5. 目标用户

### 用户画像（新增角色）

| 角色 | 卖方 | 买方 | 求购者 | 管理员 |
|------|------|------|--------|--------|
| 身份 | 大三/大四学生 | 大一/大二学生 | 各年级学生 | 学校/运营团队 |
| 动机 | 清闲置 + 赚点零花钱 + 积累好评 | 省钱淘好物 + 安全交易 | 有明确需求但找不到货源 | 维护平台秩序 |
| 频率 | 每学期 1-3 次 | 每月 2-5 次 | 每学期 2-4 次 | 每日登录 |
| 核心诉求 | 卖出快、不扯皮、好评多 | 价格低、描述真、有保障 | 有人能看到我的需求并报价 | 违规可追溯、数据可查看 |

### 使用场景（V2 新增）

5. **担保交易**：买方看中商品 → 微信支付到平台 → 双方面交 → 买方确认收货 → 平台打款卖方
6. **交易纠纷**：买方收货时发现与描述严重不符 → 拒绝确认 → 发起退款申诉 → 管理员介入
7. **求购发布**：开学前需要一本教材 → 发布求购 → 有书的卖方看到并响应报价
8. **举报处理**：用户发现违规商品 → 举报 → 管理员后台审核 → 下架/驳回

## 6. 需求范围

### 包含（In Scope）— V2.0

#### P0：交易闭环（必须有）

- [ ] **微信支付接入（模拟）**
  - 买方在商品详情页发起支付，调起微信支付（MVP 阶段模拟支付流程）
  - 预留真实微信支付 API 接口（统一下单、支付回调、退款）
  - 支付后生成订单，状态流转：待付款 → 已付款（待面交）→ 已完成 → 已取消
  - 支持买方申请退款、卖方同意/拒绝退款、管理员仲裁

- [ ] **交易订单管理**
  - 订单列表（买方视角：我买到的；卖方视角：我卖出的）
  - 订单详情：商品信息、金额、对方信息、状态时间线
  - 订单状态流转 + 状态变更通知

- [ ] **货款托管**
  - 买方付款后资金计入"平台托管"状态（虚拟账户）
  - 买方确认收货后自动解付给卖方
  - 退款场景下资金原路退回买方

#### P0：信用体系（必须有）

- [ ] **多维评价系统**
  - 交易完成后 7 天内可互评
  - 三个评价维度：沟通态度（1-5 星）、描述相符度（1-5 星）、交易速度（1-5 星）
  - 文字评价（选填，最多 200 字）
  - 综合信用分：三个维度加权计算（沟通 20% + 相符度 50% + 速度 30%）

- [ ] **信用展示**
  - 个人主页展示综合信用分 + 三维度分项
  - 商品详情页卖家头像旁展示信用分徽章
  - 评价列表可查看历史评价详情

#### P0：内容安全（必须有）

- [ ] **举报系统**
  - 商品详情页"举报"入口
  - 举报类型：虚假信息 / 违规商品 / 诈骗行为 / 其他
  - 举报描述（必填，最少 10 字）
  - 同一用户对同一商品不可重复举报

- [ ] **举报处理（管理后台）**
  - 举报列表：待处理 / 已处理
  - 处理操作：下架商品 / 驳回举报 / 封禁用户
  - 处理结果通知举报人

#### P1：求购专区（应该有）

- [ ] **求购发布**
  - 求购表单：标题、描述、预算范围（最低/最高）、分类、期望成色
  - 求购列表：按发布时间排序，支持分类筛选
  - 求购详情页

- [ ] **求购响应**
  - 卖方可对求购发起响应（留言 + 报价）
  - 响应后可转入聊天
  - 卖方可关联自己已有的商品

- [ ] **求购匹配**
  - 求购详情页下方推荐相关在售商品
  - 商品详情页提示相关求购需求

#### P1：管理后台 + 数据看板（应该有）

- [ ] **举报管理**
  - 举报列表（待处理/已处理）
  - 举报详情 + 操作按钮

- [ ] **商品管理**
  - 商品列表（支持搜索、分类筛选）
  - 手动下架/恢复操作

- [ ] **用户管理**
  - 用户列表（支持搜索学号/昵称）
  - 封禁/解封操作

- [ ] **数据看板**
  - 概览卡片：今日 GMV、今日订单、今日发布、活跃用户
  - 趋势图表：近 30 天 GMV/订单/发布/用户 折线图
  - 分类分布：商品分类占比饼图
  - 交易漏斗：浏览→咨询→付款→完成

### 不包含（Out of Scope）— 后续版本

### 依赖

## 7. 方案设计

### 7.1 整体架构

```
[浏览器] Vue3 SPA (用户端 + 管理端)
    ↕ REST API（轮询方式聊天）
[Flask 应用] (API 服务)
    ↕
[SQLite] 数据库
[本地文件系统] 图片存储
[模拟支付网关] Mock WeChat Pay
```

管理后台与用户端共用同一后端，通过路由前缀 `/admin/` 区分，管理端使用独立的管理员账号密码认证。

### 7.2 信息架构

```
V2 新增页面：

交易相关：
├── 订单列表（我买到的 / 我卖出的）
├── 订单详情（状态时间线 + 操作按钮）
├── 支付页面（确认订单 → 模拟支付）
└── 评价页面（三维度打分 + 文字评价）

求购相关：
├── 求购列表（分类筛选）
├── 求购详情（含商品推荐）
└── 发布求购页

管理后台（/admin）：
├── 后台登录
├── 仪表盘（数据看板）
├── 举报管理
├── 商品管理
└── 用户管理

V1 改造页面：
├── 商品详情页 → 新增"立即购买"按钮 + 举报入口 + 卖家信用分
├── 个人中心 → 新增信用分展示 + 评价列表
└── 个人主页 → 新增交易记录 + 信用展示
```

### 7.3 核心业务流程

#### 担保交易流程

```
买方点击"立即购买"
  → 进入订单确认页（商品信息 + 金额）
  → 点击"确认支付"
  → 调起模拟微信支付（输入"支付密码" → 支付成功）
  → 生成订单，状态 = 已付款（待面交）
  → 双方通过聊天约定面交时间地点
  → 面交完成，买方点击"确认收货"
  → 弹出确认框："确认收货后将打款给卖方，不可撤销"
  → 买方确认 → 订单状态 = 已完成
  → 货款从平台托管自动转入卖方虚拟账户
  → 双方可互相评价（7 天有效）

退款流程：
买方发起退款申请（填写原因）
  → 订单状态 = 退款中
  → 卖方收到通知 → 同意 / 拒绝
  → 卖方同意 → 退款原路返回 → 订单状态 = 已退款
  → 卖方拒绝 → 管理员介入仲裁 → 判定支持买方/卖方
```

#### 求购流程

```
买方发布求购（标题/描述/预算/分类/期望成色）
  → 求购上架，出现在求购列表
  → 卖方浏览求购列表，看到匹配的求购
  → 卖方点击"响应"，填写留言 + 报价
  → 系统通知求购者"有人响应了你的求购"
  → 双方进入聊天对话
  → 达成一致后可跳转到商品详情（卖方关联商品）或直接交易
```

#### 举报处理流程

```
用户在商品详情页点击"举报"
  → 选择举报类型 + 填写描述（≥10 字）
  → 提交举报
  → 管理员后台收到举报，状态 = 待处理
  → 管理员审核：查看商品详情 + 举报描述
  → 操作：
    - 下架商品：商品状态变为"已下架"，通知卖家
    - 驳回举报：举报状态变为"已驳回"，通知举报人
    - 封禁用户：多次违规的用户进行封禁
  → 处理完成 → 举报状态 = 已处理
```

### 7.4 数据库设计（V2 新增表）

#### 新增表一览

```
订单表 (order):
  id, order_no (唯一订单号), buyer_id (FK user), seller_id (FK user),
  product_id (FK product), amount (支付金额), status (订单状态),
  buyer_confirmed_at, seller_confirmed_at, refund_reason,
  refund_status, created_at, updated_at

支付记录表 (payment):
  id, order_id (FK order), user_id (FK user), amount,
  transaction_id (模拟交易号), status (支付状态),
  pay_type (wechat), created_at

评价表 (review):
  id, order_id (FK order), reviewer_id (FK user),
  reviewee_id (FK user), communication_score (1-5),
  description_score (1-5), speed_score (1-5),
  comment (评价文字), created_at

举报表 (report):
  id, reporter_id (FK user), product_id (FK product),
  report_type (举报类型), description (举报描述),
  status (待处理/已处理/已驳回), handler_note (处理备注),
  created_at, handled_at

求购表 (wanted):
  id, user_id (FK user), title, description,
  budget_min, budget_max, category, desired_condition,
  status (进行中/已关闭), created_at, updated_at

求购响应表 (wanted_response):
  id, wanted_id (FK wanted), responder_id (FK user),
  price_offer (报价), message, product_id (关联商品, 可选),
  created_at

管理员表 (admin):
  id, username, password_hash, role (admin),
  created_at
```

#### V1 表改动

```
用户表 (user):
  + credit_score (综合信用分, default 0)
  + credit_count (评价次数, default 0)

商品表 (product):
  + wanted_id (关联求购ID, 可为空)
  + report_count (被举报次数, default 0)
```

### 7.5 订单状态机

```
                    ┌─────────────────────────────┐
                    │                             ↓
待付款 ──支付成功──→ 已付款(待面交) ──买方确认──→ 已完成 ──互评──→ 已评价
  │                     │                         ↑
  │                     │ 买方申请退款              │
  │                     ↓                         │
  │                  退款中 ──卖方同意──→ 已退款 ───┘
  │                     │
  │                     │ 卖方拒绝
  │                     ↓
  │                  管理员仲裁
  │                  ↙        ↘
  │              已退款      已完成
  │                              ↑
  └──超时取消─────────────────────┘
```

### 7.6 异常场景处理

| 场景 | 处理方式 |
|------|----------|
| 支付时微信接口超时 | 前端显示"支付处理中"，轮询查询支付状态（3s × 10 次），超时后提示"支付状态未知，请查看订单" |
| 支付成功但订单未生成 | 支付回调幂等处理，基于 transaction_id 去重，避免重复创建订单 |
| 买方超时未确认收货 | 支付后 7 天自动确认收货，打款卖方 |
| 买方申请退款，卖方 48h 未响应 | 自动升级为管理员介入 |
| 双方串通刷好评 | 仅交易成功后可评价，每个订单仅一条评价，不支持修改 |
| 求购长时间无人响应 | 7 天后自动关闭，提示"暂无人响应，建议调整预算或描述" |
| 管理后台被未授权访问 | 管理端独立登录，JWT 中携带 admin role，普通用户 token 无法访问 /admin/* |
| 退款中商品被重新发布 | 退款期间商品状态锁定为"交易中"，不可重复购买 |

## 8. 技术要求

### 系统模块

| 模块 | 技术 | 说明 |
|------|------|------|
| 前端 SPA | Vue3 + Vue Router + Pinia | 用户端 + 管理端双路由（/ 和 /admin） |
| UI 框架 | Element Plus | 统一用户端和管理端风格 |
| 后端 API | Flask + Flask-RESTful | RESTful API，新增 /api/order、/api/review、/api/wanted、/api/report、/admin/* |
| 数据库 | SQLite + Flask-SQLAlchemy | 新增 6 张表 + V1 表字段扩展 |
| 聊天 | 轮询（维持 V1 方案） | 3s 轮询，V2 不做升级 |

## 9. 验收标准

### 担保交易

- [ ] Given 买方在商品详情页，When 点击"立即购买"并完成模拟支付，Then 订单生成且状态为"已付款"
- [ ] Given 订单已付款，When 买方点击"确认收货"，Then 弹出不可撤销提示，确认后订单状态变为"已完成"
- [ ] Given 订单已付款，When 买方发起退款申请并填写原因，Then 订单状态变为"退款中"，卖方收到退款通知
- [ ] Given 退款申请待处理，When 卖方同意退款，Then 订单状态变为"已退款"，金额退回买方虚拟账户
- [ ] Given 退款申请待处理，When 卖方拒绝退款，Then 订单升级为"管理员介入"状态
- [ ] Given 订单已付款超过 7 天，When 买方未操作，Then 系统自动确认收货

### 信用评价

- [ ] Given 订单已完成后 7 天内，When 买方进入评价页并对三个维度打分+写评价，Then 评价发布成功，卖方信用分更新
- [ ] Given 已完成交易的用户，When 他人查看其个人主页，Then 看到综合信用分和三维度评分
- [ ] Given 已评价的订单，When 用户尝试再次评价同一订单，Then 提示"已评价过该订单"
- [ ] Given 未完成交易，When 用户进入评价入口，Then 提示"交易完成后才能评价"

### 举报

- [ ] Given 用户在商品详情页，When 点击举报并选择类型+填写描述（≥10字），Then 举报提交成功
- [ ] Given 同一用户对同一商品，When 尝试再次举报，Then 提示"你已举报过该商品"
- [ ] Given 管理员在后台，When 查看待处理举报并点击"下架商品"，Then 商品状态变为"已下架"并通知卖家

### 求购专区

- [ ] Given 已登录用户，When 填写求购信息（标题+描述+预算范围）并发布，Then 求购上架成功
- [ ] Given 求购列表，When 用户切换分类筛选，Then 按分类过滤
- [ ] Given 卖方在求购详情页，When 填写报价+留言并提交响应，Then 求购者收到通知并可进入聊天
- [ ] Given 求购发布超过 7 天无人响应，Then 系统自动将该求购状态设为"已关闭"

### 管理后台

- [ ] Given 管理员登录，When 输入正确账号密码，Then 进入管理后台仪表盘
- [ ] Given 普通用户 token，When 访问 /admin/* 路由，Then 返回 403 或重定向
- [ ] Given 管理后台仪表盘，When 查看数据看板，Then 展示 GMV/订单/发布/用户概览卡片 + 趋势图表

### 边界与异常

- [ ] Given 支付过程中网络断开，When 用户重新进入，Then 可查看订单状态并重新支付（如未支付成功）
- [ ] Given 卖方被封禁，When 买方对其已付款订单确认收货，Then 交易正常完成，货款解付
- [ ] Given 商品已进入交易中（有未完成订单），When 其他用户尝试购买，Then 提示"该商品已被预订"
- [ ] Given 管理员下架商品后，When 已有用户在其聊天中，Then 聊天可继续但商品详情显示"已下架"

## 10. 附录

### 竞品分析（V2 相关）

| 竞品 | 支付方案 | 评价体系 | 求购功能 | 对我们的启示 |
|------|---------|---------|---------|------------|
| 闲鱼 | 支付宝担保交易，全款托管 | 单维度星级 + 文字评价 | 有"求购/想要"功能 | V2 交易流程参考闲鱼简化版 |
| 转转 | 微信支付担保 | 单维度 + 标签 | 无求购 | — |
| 校内微信群 | 线下转账 | 群内口碑（非结构化） | 发消息求购（效率低） | 求购是刚需但现有方案极低效 |
| 各校自建二手平台 | 多数无支付 | 多数无评价 | 少数有简版 | V2 的差异化在于完整交易闭环 |

### 技术选型补充说明

- **为什么继续用 SQLite**：V2 预估数据量（万级订单、十万级消息）仍在 SQLite 舒适区。后续如需 MySQL 迁移，SQLAlchemy ORM 层只需改连接字符串
- **为什么用模拟支付**：真实微信支付需要企业资质+商户号+对公账户，学生项目不具备条件。Mock 接口保留与真实 API 相同的请求/响应格式，后续接入时只需替换 URL 和密钥
- **为什么不做 WebSocket**：V1 已验证 3s 轮询方案在千级并发下表现稳定，V2 范围已足够大，WebSocket 改造留到 V2.1
