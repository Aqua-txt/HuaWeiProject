## 2. 背景与动机

### 市场背景

高校校园内二手交易需求旺盛，但现有渠道（微信群、QQ群、贴吧）存在信息分散、不可搜索、无法保障交易安全等核心痛点。闲鱼等通用平台缺少校园社交信任背书，校内自建平台多数停留在信息发布层面，缺乏交易闭环。

### 产品演进

| 版本 | 核心能力 | 解决的问题 |
|------|---------|-----------|
| V1.0 | 信息撮合（发布/浏览/搜索/聊天/收藏） | 信息分散、不可搜索 |
| V2.0 | 交易闭环 + 信用体系 + 求购专区 + 管理后台 | 交易无保障、无信用积累、供给侧单一、无运营抓手 |
| V3.0 | 全功能完善 + 工程加固 | 功能完整性、健壮性、体验打磨 |

### V3.0 的目标

将项目从"功能可用"升级为"功能完备且健壮"，确保每一个已实现功能都有完整的产品定义、验收标准和异常处理，消除功能盲区和体验断点。

## 3. 问题定义

### 核心问题

校园二手交易在**信息获取、交易安全、信任判断、需求匹配、平台治理**五个维度存在系统性缺陷，导致交易效率低、用户粘性弱、平台价值停留于浅层。

### 痛点全景

| 维度 | 角色 | 痛点 | 场景 |
|------|------|------|------|
| 信息获取 | 买方 | 商品分散在多个群，无法统一搜索 | 想买一本高数教材，翻遍5个群找不到 |
| 信息获取 | 买方 | 不知道商品成色、价格是否合理 | 图片模糊、描述简单，无法判断 |
| 交易安全 | 买方 | 线下转账无保障，货不对板无法追责 | 先转账后见面，发现商品与描述不符 |
| 交易安全 | 卖方 | 遇到恶意退款/纠纷无法自证 | 买家收到货后恶意申请退款 |
| 信任判断 | 双方 | 每次交易都是"一次性博弈" | 好卖家无法积累信誉，差卖家没有代价 |
| 需求匹配 | 买方 | 想买的特定物品没人发布 | 冷门教材/配件，列表中找不到 |
| 需求匹配 | 卖方 | 不知道买家愿意出多少钱 | 二手笔记本定价困难 |
| 平台治理 | 管理员 | 违规内容靠用户反馈，缺乏主动管理 | 不知道多少商品违规、多少交易产生纠纷 |
| 平台治理 | 管理员 | 缺乏数据支撑决策 | 不清楚平台GMV、用户活跃度、交易转化率 |

### 用户价值主张

**"让每一笔校园交易都有保障、有记录、有信用——像当面交易一样安全，像线上平台一样高效。"**

## 4. 目标与指标

### 产品目标

构建"信息发现 → 担保交易 → 信用评价 → 双向匹配 → 运营治理"的完整服务闭环，让用户不仅能"找到货"，还能"放心交易"并"找到人"。

### 成功指标

| 指标类型 | 指标名称 | 目标值 | 说明 |
|----------|----------|--------|------|
| **北极星指标** | 月担保交易笔数 | ≥ 200 笔/月 | 衡量交易闭环是否被用户接受 |
| **北极星指标** | 交易完成率 | ≥ 70%（付款→完成） | 衡量交易流程是否顺畅 |
| **过程指标** | 支付发起率 | ≥ 商品咨询量的 20% | 支付入口的转化效率 |
| **过程指标** | 评价覆盖率 | ≥ 完成交易的 60% | 评价体系是否被用户使用 |
| **过程指标** | 求购发布数 | ≥ 100 条/月 | 求购专区的活跃度 |
| **过程指标** | 求购匹配率 | ≥ 30%（有人响应） | 求购需求的满足效率 |
| **过程指标** | DAU | ≥ 500 | 日活跃用户数 |
| **过程指标** | 搜索使用率 | ≥ 40% | 搜索功能的渗透度 |
| **过程指标** | 收藏转化率 | ≥ 15%（收藏→下单） | 收藏对交易的促进作用 |
| **反向指标** | 交易纠纷率 | ≤ 5%（申诉/退款笔数） | 交易质量底线 |
| **反向指标** | 违规商品存活时间 | ≤ 24 小时（从发布到举报处理） | 先发后审的风险窗口 |
| **管理指标** | 举报处理时效 | ≤ 12 小时（工作日） | 运营响应速度 |

## 5. 目标用户

### 用户画像

| 角色 | 身份 | 动机 | 使用频率 | 核心诉求 |
|------|------|------|---------|---------|
| 卖方 | 大三/大四学生 | 清闲置 + 赚零花钱 + 积累好评 | 每学期 1-3 次 | 卖出快、不扯皮、好评多 |
| 买方 | 大一/大二学生 | 省钱淘好物 + 安全交易 | 每月 2-5 次 | 价格低、描述真、有保障 |
| 求购者 | 各年级学生 | 有明确需求但找不到货源 | 每学期 2-4 次 | 有人能看到需求并报价 |
| 管理员 | 学校/运营团队 | 维护平台秩序 | 每日登录 | 违规可追溯、数据可查看 |

### 使用场景

1. **信息浏览**：学生打开首页 → 浏览/搜索/筛选商品 → 查看详情 → 收藏或联系卖家
2. **商品发布**：毕业季学生 → 发布闲置物品（拍照+描述+定价）→ 等待买家咨询
3. **担保交易**：买方看中商品 → 微信支付到平台 → 双方面交 → 买方确认收货 → 平台打款卖方
4. **交易纠纷**：买方收货时发现与描述严重不符 → 拒绝确认 → 发起退款申诉 → 卖方/管理员处理
5. **信用评价**：交易完成 → 互评三维度 → 信用分积累 → 影响后续交易信任
6. **求购发布**：开学前需要教材 → 发布求购 → 卖方看到并响应报价 → 双方交易
7. **举报处理**：用户发现违规商品 → 举报 → 管理员后台审核 → 下架/封禁
8. **运营管理**：管理员登录后台 → 查看数据看板 → 处理举报 → 管理商品/用户

## 6. 需求范围

### 包含（In Scope）— V3.0 完整功能

#### P0：用户系统（必须有）

- [x] **用户注册**：学号（4-20位，唯一）+ 昵称（≤50字）+ 密码（≥6位）+ 确认密码，注册成功自动登录
- [x] **用户登录**：学号 + 密码登录，JWT token 认证（72小时有效期），密码哈希校验
- [x] **获取当前用户信息**：根据 token 获取登录用户完整信息
- [x] **编辑个人资料**：修改昵称（≤50字）、手机号、邮箱，上传头像（支持 png/jpg/jpeg/gif/webp，Pillow 自动压缩至 1200×1200，quality=85）
- [x] **个人中心展示**：头像、昵称、学号、校友标识、手机号/邮箱绑定状态，菜单入口（我的发布/我的收藏/编辑资料/退出登录）

#### P0：商品系统（必须有）

- [x] **商品发布**：标题（≤100字）、描述、价格（0.01-99999.99）、分类（10种：教材/数码/生活/其他/数码电子/图书教材/生活用品/服装配饰/运动户外/美妆护肤/食品零食）、成色（4种：全新/几乎全新/轻微使用痕迹/明显使用痕迹）、图片上传（1-6张，支持5种格式，单张≤5MB）
- [x] **商品编辑**：卖家修改已发布商品的所有字段及替换图片
- [x] **商品上架/下架**：卖家切换商品状态，下架后不出现在列表中
- [x] **我的发布管理**：查看当前用户发布的商品，按状态（在售/已下架）筛选，分页
- [x] **商品列表浏览**：分页展示上架商品，支持分类筛选、关键词搜索（标题+描述模糊匹配）、排序（最新发布/价格升序/价格降序），登录用户自动标记收藏状态
- [x] **商品详情查看**：图片轮播（触摸滑动+指示器+计数器）、卖家信息（头像/昵称/认证标签/信用分）、商品完整信息、收藏状态、底部操作栏（收藏/举报/联系卖家/立即购买）

#### P0：收藏系统（必须有）

- [x] **收藏/取消收藏**：切换商品收藏状态（已收藏则取消，未收藏则添加）
- [x] **我的收藏列表**：查看已收藏商品瀑布流展示，含收藏时间，分页

#### P0：聊天系统（必须有）

- [x] **创建会话**：基于商品发起与卖家的私信会话，不能和自己聊天，同一买家+商品+卖家复用已有会话
- [x] **会话列表**：展示所有会话（对方头像/昵称、最后一条消息预览、未读消息角标、关联商品信息及缩略图），按更新时间倒序，定时刷新
- [x] **聊天消息收发**：消息列表（区分自己/对方气泡），发送文本消息，轮询获取新消息（after_id 增量拉取），自动标记对方消息为已读
- [x] **未读消息计数**：获取未读消息总数，导航栏角标实时显示

#### P0：交易闭环（必须有）

- [x] **创建订单**：买家对上架商品下单，商品自动下架，同一买家不能对同一商品重复下单（存在待付款/已付款订单时拒绝）
- [x] **模拟支付**：微信支付模拟，生成交易流水号，订单变为已付款，确保商品下架
- [x] **取消订单**：买家取消待付款订单，商品重新上架
- [x] **确认收货**：买家确认收货，订单变为已完成，货款打给卖家
- [x] **订单列表**：按角色（我买到的/我卖出的）查看订单，分页
- [x] **订单详情**：订单号、状态、时间线、商品信息、金额明细、交易双方信息、操作按钮、退款信息、评价入口

#### P0：退款流程（必须有）

- [x] **申请退款**：买家对已付款订单申请退款，退款原因≥10字，订单变为退款中
- [x] **卖家响应退款**：卖家同意退款（生成退款支付记录，商品重新上架）或拒绝退款（订单进入管理员仲裁状态）

#### P0：信用体系（必须有）

- [x] **评价交易伙伴**：对已完成订单评价，三项评分（沟通态度/描述相符/交易速度，1-5分）+ 评语（≤200字），每人每订单仅可评价一次，7天有效期
- [x] **信用分计算**：加权公式——综合信用分 = 沟通评分 × 0.2 + 描述评分 × 0.5 + 速度评分 × 0.3，评价后自动更新被评价者信用分
- [x] **信用展示**：商品详情页卖家头像旁展示信用分徽章，个人中心展示信用分
- [x] **评价列表查看**：按评价者或被评价者查询评价列表，分页
- [x] **检查订单可评价性**：判断订单是否可评价，返回状态及原因（未完成/无权/已评价/已过期）

#### P0：内容安全（必须有）

- [x] **举报商品**：选择举报类型（虚假信息/违规商品/诈骗行为/其他）+ 描述（≥10字），同一用户同一商品不可重复举报，商品举报计数+1
- [x] **我的举报列表**：查看自己提交的举报记录，支持状态筛选，分页

#### P1：求购专区（应该有）

- [x] **发布求购**：标题（≤100字）、描述（≥10字）、预算范围（最低-最高）、分类（10种）、期望成色，默认进行中状态
- [x] **求购列表**：浏览所有求购信息，云标签式布局，支持分类筛选、状态筛选、仅看自己的，分页
- [x] **求购详情**：发布者信息、求购完整信息、响应记录列表、智能推荐商品
- [x] **智能推荐**：求购详情页推荐同类目+价格在预算范围内的上架商品（最多5个）
- [x] **编辑求购**：修改自己发布的求购信息，可修改所有字段及关闭求购
- [x] **删除求购**：删除自己发布的求购，级联删除所有响应
- [x] **响应求购**：填写报价+留言+可选关联自己的商品，不能响应自己的求购，已关闭的求购不能响应
- [x] **求购响应列表**：查看求购的所有响应记录

#### P1：管理后台（应该有）

- [x] **管理员登录**：用户名+密码登录，独立 JWT token（role=admin），72小时有效期
- [x] **管理员初始化**：首次创建管理员账号（默认 admin/admin123），已有则拒绝
- [x] **数据看板**：概览卡片（今日订单/发布/新用户，商品/用户/订单总数，已付款/已完成订单），近30天趋势图表（订单/发布/新用户），分类分布统计

- [x] **商品管理**：全平台商品列表，关键词搜索、分类筛选、状态筛选，管理员强制上架/下架
- [x] **用户管理**：全平台用户列表，关键词搜索（学号/昵称），查看信用分/评价数/状态，封禁/解封

#### P1：系统基础设施（应该有）

- [x] **JWT 双通道认证**：普通用户 token（user_id）+ 管理员 token（admin_id+role=admin），HS256 签名，72小时过期，login_required/optional_login/admin_required 三种装饰器
- [x] **图片上传与处理**：UUID 重命名、Pillow 压缩（1200×1200，quality=85，optimize=True）、格式校验、最大 32MB
- [x] **CORS 跨域**：Flask-CORS 配置，允许前端 localhost:5173 和 127.0.0.1:5173
- [x] **Vite 开发代理**：/api 请求代理到后端 127.0.0.1:5000，支持环境变量 VITE_BACKEND_PROXY_TARGET
- [x] **Pinia 状态管理**：token+user 对象存储在 localStorage，isLoggedIn 计算属性，setAuth/fetchUser/updateUser/logout 方法
- [x] **Axios 拦截器**：请求自动附加 Bearer token，响应自动提示成功消息（非GET），错误友好提示（网络异常/后端未启动），401 自动跳转登录页
- [x] **路由守卫**：requiresAuth 检查 token 跳转 /login，requiresAdmin 检查 admin_token 跳转 /admin/login，自动设置页面标题
- [x] **北京时间工具**：beijing_now() 函数返回 UTC+8 的 naive datetime，用于所有时间字段
- [x] **种子数据生成**：自动生成测试用户、商品、求购、订单、评价数据，含信用分计算
- [x] **数据库自动初始化**：Flask 启动时 db.create_all() 自动创建所有表

## 7. 方案设计

### 7.1 整体架构

```
[浏览器] Vue3 SPA (用户端 + 管理端)
    ↕ REST API（轮询方式聊天，3s 间隔）
[Flask 应用] (API 服务)
    ↕
[SQLite] 数据库（campus_trade.db）
[本地文件系统] 图片存储（uploads/）
[模拟支付网关] Mock WeChat Pay
```

管理后台与用户端共用同一后端，通过路由前缀 `/admin/` 区分，管理端使用独立的管理员账号密码认证。

### 7.2 信息架构

```
用户端：
├── 首页 (/home) — 商品发现、搜索、筛选、排序、分类
├── 登录 (/login) — 学号+密码登录
├── 注册 (/register) — 学号+昵称+密码注册
├── 商品详情 (/product/:id) — 图片轮播+卖家信息+操作栏
├── 发布/编辑商品 (/publish, /edit-product/:id)
├── 个人中心 (/profile) — 资料展示+菜单入口
├── 编辑资料 (/edit-profile) — 头像+昵称+手机+邮箱
├── 我的发布 (/my-products) — 在售/已下架 Tab
├── 我的收藏 (/my-favorites) — 瀑布流展示
├── 消息列表 (/messages) — 会话列表+未读角标
```

### 7.3 核心业务流程

#### 商品交易全流程

```
卖方发布商品（填写信息+上传图片）
  → 商品上架，出现在首页列表
  → 买方浏览/搜索/筛选 → 查看详情
  → 买方点击"立即购买"
  → 生成订单，状态 = 待付款，商品自动下架
  → 买方进入支付页 → 确认订单 → 模拟微信支付
  → 支付成功，订单状态 = 已付款（待面交），生成支付记录
  → 双方通过聊天约定面交时间地点
  → 面交完成，买方点击"确认收货"
  → 弹出不可撤销提示，确认后订单状态 = 已完成
  → 货款从平台托管自动转入卖方虚拟账户
  → 双方可互相评价（7天内有效）
  → 评价后更新信用分
```

#### 退款流程

```
买方发起退款申请（填写原因，≥10字）
  → 订单状态 = 退款中
  → 卖方收到通知 → 同意 / 拒绝
  → 卖方同意 → 生成退款支付记录，商品重新上架 → 订单状态 = 已退款
  → 卖方拒绝 → 订单状态 = 管理员仲裁
```

#### 求购流程

```
买方发布求购（标题/描述/预算范围/分类/期望成色）
  → 求购上架，出现在求购列表
  → 求购详情页展示智能推荐商品（同类目+价格在预算范围内的上架商品，最多5个）
  → 卖方浏览求购列表，看到匹配的求购
  → 卖方点击"响应"，填写报价+留言+可选关联自己商品
  → 响应记录出现在求购详情页
  → 双方可进入聊天对话
```

#### 举报处理流程

```
用户在商品详情页点击"举报"
  → 选择举报类型（虚假信息/违规商品/诈骗行为/其他）+ 填写描述（≥10字）
  → 提交举报，商品举报计数+1
  → 管理员后台收到举报，状态 = 待处理
  → 管理员审核：
    - 驳回举报：举报状态 = 已驳回
    - 下架商品：商品状态 = 已下架
    - 下架+封禁用户：商品下架 + 用户封禁
  → 处理完成，举报状态 = 已处理
```

### 7.7 异常场景处理

| 场景 | 处理方式 |
|------|----------|
| 支付时模拟接口超时 | 前端显示"支付处理中"，提示用户查看订单状态 |
| 支付成功但订单未生成 | 支付回调幂等处理，基于 transaction_id 去重 |
| 买方超时未确认收货 | 需手动确认（V3.0 暂不实现自动确认，V3.1+ 加入 7天自动确认） |
| 买方申请退款，卖家拒绝 | 订单进入"管理员仲裁"状态，管理员后台处理 |
| 双方串通刷好评 | 仅交易成功后可评价，每个订单仅一条评价，不支持修改 |
| 求购长时间无人响应 | 需手动关闭（V3.0 暂不实现自动关闭，V3.1+ 加入 7天自动关闭） |
| 管理后台被未授权访问 | 管理端独立登录，JWT 中携带 admin role，普通用户 token 无法访问 /admin/* |
| 退款中商品被重新发布 | 退款期间商品状态锁定为"交易中"（下单时自动下架） |
| 卖家被封禁 | 已有订单可正常完成，封禁后不可发布新商品 |
| 后端服务未启动 | 前端 Axios 错误拦截器友好提示"服务器开小差了，请稍后再试" |
| Token 过期 | 401 响应 → 前端自动跳转登录页，清除本地存储 |

## 9. 验收标准

### 用户系统

- [ ] Given 未注册用户在注册页，When 填写有效学号+昵称+密码并提交，Then 注册成功并自动登录，跳转首页
- [ ] Given 学号已存在的用户，When 尝试注册相同学号，Then 提示"该学号已注册"
- [ ] Given 已注册用户在登录页，When 输入正确学号+密码，Then 登录成功，token 存入 localStorage
- [ ] Given 登录用户，When 输入错误密码，Then 提示"学号或密码错误"
- [ ] Given 登录用户在编辑资料页，When 上传新头像并保存，Then 头像更新成功（图片已压缩）
- [ ] Given 登录用户在编辑资料页，When 修改昵称为超过50字，Then 提示字数限制

### 商品系统

- [ ] Given 登录用户在发布页，When 填写完整商品信息+上传1-6张图片并提交，Then 商品发布成功并出现在首页
- [ ] Given 登录用户在发布页，When 未上传图片或标题为空，Then 提示必填项
- [ ] Given 登录用户在商品详情页，When 点击"编辑商品"，Then 进入编辑页，可修改所有字段
- [ ] Given 卖家在商品详情页，When 点击"下架"，Then 商品状态变为"已下架"，不再出现在首页列表
- [ ] Given 任意用户在首页，When 输入关键词搜索，Then 返回标题或描述含关键词的商品
- [ ] Given 任意用户在首页，When 选择分类筛选，Then 仅显示该分类商品
- [ ] Given 任意用户在首页，When 选择"价格升序"排序，Then 商品按价格从低到高排列

### 收藏系统

- [ ] Given 登录用户在商品详情页，When 点击收藏按钮，Then 商品加入收藏列表
- [ ] Given 已收藏商品的用户，When 再次点击收藏按钮，Then 取消收藏
- [ ] Given 登录用户在我的收藏页，When 查看收藏列表，Then 瀑布流展示所有已收藏商品

### 聊天系统

- [ ] Given 登录用户在商品详情页，When 点击"联系卖家"，Then 创建会话并跳转聊天页
- [ ] Given 登录用户在聊天页，When 输入消息并点击发送，Then 消息出现在聊天列表
- [ ] Given 聊天中对方发送新消息，When 3s 轮询拉取，Then 新消息出现且标记为已读
- [ ] Given 有未读消息的用户，When 查看导航栏，Then 消息图标显示未读数角标
- [ ] Given 用户尝试与自己聊天，When 创建会话，Then 提示"不能和自己聊天"

### 交易闭环

- [ ] Given 买方在商品详情页，When 点击"立即购买"，Then 生成订单且状态为"待付款"，商品自动下架
- [ ] Given 买方在支付页，When 确认支付，Then 订单状态变为"已付款"，生成支付记录
- [ ] Given 订单已付款，When 买方点击"确认收货"，Then 弹出不可撤销提示，确认后订单状态变为"已完成"
- [ ] Given 订单待付款，When 买方点击"取消订单"，Then 订单状态变为"已取消"，商品重新上架
- [ ] Given 同一买方对同一商品已有待付款/已付款订单，When 再次购买，Then 提示"您已购买该商品"
- [ ] Given 已下架商品，When 买方尝试购买，Then 提示商品不可购买

### 退款流程

- [ ] Given 订单已付款，When 买方发起退款申请并填写原因（≥10字），Then 订单状态变为"退款中"
- [ ] Given 退款申请待处理，When 卖方同意退款，Then 订单状态变为"已退款"，商品重新上架
- [ ] Given 退款申请待处理，When 卖方拒绝退款，Then 订单状态变为"管理员仲裁"

### 信用评价

- [ ] Given 订单已完成后7天内，When 买方进入评价页并对三个维度打分+写评价，Then 评价发布成功，卖方信用分更新
- [ ] Given 信用分计算，When 新评价提交，Then 被评价者信用分 = (历史总分 + 新评分加权) / 评价次数
- [ ] Given 已评价的订单，When 用户尝试再次评价，Then 提示"已评价过该订单"
- [ ] Given 未完成交易，When 用户进入评价入口，Then 提示"交易完成后才能评价"
- [ ] Given 评价超过7天有效期，When 用户尝试评价，Then 提示"评价已过期"

### 举报功能

- [ ] Given 用户在商品详情页，When 点击举报并选择类型+填写描述（≥10字），Then 举报提交成功
- [ ] Given 同一用户对同一商品，When 尝试再次举报，Then 提示"你已举报过该商品"
- [ ] Given 用户在我的举报列表，When 查看举报记录，Then 显示举报状态（待处理/已处理/已驳回）

### 求购专区

- [ ] Given 已登录用户，When 填写求购信息（标题+描述+预算范围）并发布，Then 求购上架成功
- [ ] Given 求购列表，When 用户切换分类筛选，Then 按分类过滤
- [ ] Given 卖方在求购详情页，When 填写报价+留言并提交响应，Then 响应记录出现在求购详情
- [ ] Given 求购详情页，When 查看推荐商品，Then 展示同类目+价格在预算范围内的上架商品（最多5个）
- [ ] Given 求购发布者，When 编辑求购信息，Then 求购信息更新成功
- [ ] Given 求购发布者，When 删除求购，Then 求购及其所有响应被删除
- [ ] Given 用户尝试响应自己的求购，When 提交响应，Then 提示"不能响应自己的求购"

### 管理后台

- [ ] Given 管理员在登录页，When 输入正确账号密码，Then 登录成功进入管理后台
- [ ] Given 首次访问管理后台，When 无管理员账号，Then 可初始化管理员（默认 admin/admin123）
- [ ] Given 普通用户 token，When 访问 /admin/* 路由，Then 返回 403 或重定向到管理员登录页
- [ ] Given 管理后台仪表盘，When 查看数据看板，Then 展示概览卡片 + 趋势图表 + 分类分布
- [ ] Given 管理员在举报管理页，When 查看待处理举报并点击"下架商品"，Then 商品状态变为"已下架"
- [ ] Given 管理员在举报管理页，When 点击"下架+封禁"，Then 商品下架且用户被封禁
- [ ] Given 管理员在商品管理页，When 搜索商品关键词，Then 返回匹配的商品列表
- [ ] Given 管理员在用户管理页，When 封禁用户，Then 用户状态变为"封禁"，无法登录
- [ ] Given 管理员在用户管理页，When 解封用户，Then 用户恢复正常状态

### 边界与异常

- [ ] Given 支付过程中网络断开，When 用户重新进入，Then 可查看订单状态
- [ ] Given 卖方被封禁，When 买方对其已付款订单确认收货，Then 交易正常完成，货款解付
- [ ] Given 商品已进入交易中（有未完成订单），When 其他用户尝试购买，Then 提示"该商品已被预订"
- [ ] Given 管理员下架商品后，When 已有用户在其聊天中，Then 聊天可继续但商品详情显示"已下架"
- [ ] Given 后端服务未启动，When 前端发起请求，Then 友好提示"服务器开小差了，请稍后再试"
- [ ] Given Token 过期，When 前端发起请求收到 401，Then 自动跳转登录页并清除本地存储
- [ ] Given 图片上传超过 5MB，When 提交表单，Then 提示"图片大小不能超过5MB"
- [ ] Given 图片格式不支持，When 上传 .bmp 等格式，Then 提示支持的格式列表
