# 路演大纲：校园二手交易平台

> 用于团队评审会议，每节约 1-3 张幻灯片篇幅。建议转制 PPT 时保持此结构。

---

## 1. 一页摘要

| 项目 | 内容 |
|------|------|
| **项目名称** | 校园二手交易平台 |
| **一句话概述** | 一个让本校学生发布和搜索二手物品、并直接沟通交易的校园专属平台 |
| **技术栈** | Vue3 + Flask + SQLite |
| **预期交付时间** | 4-5 周（1 人全栈） |
| **请求决策事项** | ① 是否采用学校邮箱验证作为 MVP 身份验证方案？② 首批推广渠道如何获取种子用户？ |

---

## 2. 背景与机会

### 当前状态

- 校内二手交易目前完全依赖 **微信群/QQ群/表白墙** 等非结构化渠道
- 信息零散、不可搜索、易被刷屏，成交率极低
- 校外平台（闲鱼）非校园专属，缺乏面对面交易的信任基础

### 机会点

- **高频刚需**：每学期开学（买教材）+ 期末/毕业季（卖闲置）两个爆发节点
- **信任优势**：校园环境天然具备信任基础，同校当面交易免去邮寄和欺诈顾虑
- **零成本获客**：校内推广渠道集中（班级群、公众号、食堂海报），获客成本极低

### 关键数据

- 假设校区 2 万在校生，按 15% MAU 目标 = 3,000 月活用户
- 保守预估每月产生 ≥ 500 件商品发布
- 同类校园二手产品（如各校自建平台）的平均搜索使用率达发布量的 3 倍以上

### 不做的代价

- 学生继续在微信群刷屏式交易，体验持续恶化
- 校内闲置物品流通效率低下，资源浪费
- 失去以「交易」为切入点建立校园生活服务生态的机会窗口

---

## 3. 方案概览

### 核心架构

```
[桌面浏览器] Vue3 + Element Plus SPA
        ↕ REST API + WebSocket
    [Flask 后端 (Python)]
        ↕
 [SQLite 数据库]  [本地图片存储]
```

### 核心体验流程

```
用户注册（学校邮箱验证）
    ↓
浏览首页商品信息流 ──关键词搜索→ 搜索结果
    ↓                              ↓
点击商品 → 商品详情页 → 「联系卖家」→ 即时聊天
    ↓
达成一致 → 校内当面交易（平台外完成）
```

### 为什么是这个方案

| 候选方案 | 评价 | 原因 |
|----------|------|------|
| Vue3 + Flask + SQLite | ✅ **选定** | 轻量快速，MVP 阶段开发效率最高，SQLite 零运维 |
| Vue3 + Flask + MySQL | ❌ 暂不选 | MVP 阶段数据量级（万级）下 SQLite 完全够用，无需额外部署 |
| React + Django + PostgreSQL | ❌ 暂不选 | 团队技术栈偏好和上手成本高于 Vue3 + Flask |
| 微信小程序 | ❌ 暂不选 | 小程序审核周期长，且桌面端管理不便，开发调试效率低 |

## 4. 预期影响

### 业务指标预估

| 指标 | 当前水平 | 上线 3 个月目标 | 预估依据 |
|------|---------|---------------|---------|
| 月活跃用户 | 0（无平台） | ≥ 在校生 15% | 同类校园产品首季度 MAU 通常在 10-20% |
| 商品发布数 | 0（分散在各群） | ≥ 500 件/月 | 按活跃用户 15% 的发布转化估算 |
| 搜索使用率 | 0 | ≥ 1,500 次/月 | 搜索是核心流量分发入口，预期为发布数的 3 倍 |

### 投入产出分析

| 投入项 | 预估 |
|--------|------|
| 研发投入 | 16-23 人日（约 3-4 周） |
| 服务器成本 | 极低（SQLite + 本地存储，初期 VPS 即可） |
| 运营成本 | 几乎为零（校园渠道推广，无投放费用） |
| 预期收益 | 提升校园闲置物品流通效率，积累学生用户为后续校园服务生态打基础 |

## 5. 里程碑与排期

| 里程碑 | 时间 | 交付物 |
|--------|------|--------|
| **需求定稿** | Day 1 | 已评审 PRD |
| **前端框架搭建** | Day 2-4 | Vue3 项目骨架、路由、登录注册页 |
| **用户系统** | Day 5-8 | 注册/登录/个人中心前后端 |
| **商品系统** | Day 9-14 | 发布/编辑/删除/列表/详情前后端 |
| **搜索与筛选** | Day 15-17 | 关键词搜索 + 分类筛选 |
| **聊天系统** | Day 18-23 | 即时聊天前后端（WebSocket） |
| **联调与测试** | Day 24-27 | 全流程联调、边界测试 |
| **上线** | Day 28 | 部署、推广 |

### MVP 范围

如果时间紧张，可按以下顺序裁剪：

1. **不可砍**：用户系统 + 商品发布 + 商品浏览/搜索 + 聊天（四个支柱功能）
2. **可简化**：聊天从 WebSocket 降级为短轮询（减少开发量约 2 天）
3. **可延后**：收藏功能、商品编辑（发布后可先下架重发）

## 6. 风险与备选

### 风险矩阵

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| SQLite 并发性能不足 | 低 | 中 | MVP 阶段用户量小，SQLite 足够；如有瓶颈可升级 SQLite→MySQL |
| 学校邮箱验证被拦截 | 中 | 中 | 提供验证码重发机制，同时支持管理员手动激活账号 |
| 聊天 WebSocket 不稳定 | 中 | 低 | 降级方案为 3 秒短轮询，用户无感知切换 |
| 用户发布虚假/违规商品 | 中 | 高 | 上线举报功能 + 人工审核下架机制 |
| 无人发布，冷启动失败 | 高 | 高 | 运营策略：团队先发布 20-30 条模拟商品营造氛围 + 班级群定向邀请 |

### 需要决策的事项

1. **运营策略**：冷启动期是否由运营团队先发布一批模拟商品来激活氛围？
2. **推广渠道**：首批种子用户通过班级群转发还是线下海报扫码？
3. **命名方案**：平台名称是否需要与学校品牌关联？（如"XX 集市" vs 独立品牌名）
