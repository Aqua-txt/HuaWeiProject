# 团队协作开发指南

> 本文档指导团队成员如何基于 Git 进行标准的分支开发与 Pull Request 协作流程。

---

## 1. 前置准备

### 克隆仓库

```bash
git clone <仓库地址>
cd HuaWeiProject
```

### 配置用户信息

```bash
git config user.name "你的姓名"
git config user.email "你的邮箱"
```

### 拉取最新代码

每次开始工作前，确保本地 master 是最新的：

```bash
git checkout master
git pull origin master
```

---

## 2. 分支命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 新功能 | `feature/<功能简述>` | `feature/payment-system` |
| Bug 修复 | `fix/<问题简述>` | `fix/chat-duplicate-msg` |
| 文档更新 | `docs/<内容简述>` | `docs/api-reference` |
| 重构 | `refactor/<范围简述>` | `refactor/product-model` |

规则：
- 全小写英文，单词用 `-` 连接
- 简短且语义明确，一眼看出在做什么
- 不要使用中文、空格或特殊字符

## 3. 开发流程

### 3.1 从 master 创建功能分支

```bash
git checkout master
git pull origin master
git checkout -b feature/your-feature-name
```

### 3.2 在分支上开发

保持提交小而聚焦，每次提交做一件事：

```bash
git add <相关文件>
git commit -m "feat: 添加支付订单状态机"
```

### 3.3 提交信息规范

采用 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>: <简短描述>

<详细说明（可选）>
```

常用 type：

| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 重构（不新增功能也不修 bug） |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖变更 |

示例：

```bash
git commit -m "feat: 新增求购专区列表页"
git commit -m "fix: 修复聊天消息重复入库问题"
git commit -m "docs: 更新 API 接口文档"
```

### 3.4 同步上游代码

如果开发周期较长，定期将 master 的新提交合并到自己的分支，避免最终合并时产生大量冲突：

```bash
# 方式一：merge（推荐，保留完整历史）
git checkout master
git pull origin master
git checkout feature/your-feature-name
git merge master

# 方式二：rebase（保持线性历史，但需谨慎）
git checkout feature/your-feature-name
git fetch origin master
git rebase origin/master
```

> 注意：如果分支已经推送到远程，不要使用 rebase，否则会破坏他人的历史。

### 3.5 推送分支到远程

```bash
git push -u origin feature/your-feature-name
```

后续推送可以简化为：

```bash
git push
```

## 4. Pull Request 流程

### 4.1 创建 PR 前的自查清单

在发起 PR 之前，确保：

- [ ] 代码已经过本地测试，功能正常
- [ ] 前端 `npx vite build` 构建无错误
- [ ] 后端 `python app.py` 启动正常
- [ ] 提交信息符合规范
- [ ] 已合并 master 最新代码，无冲突
- [ ] 没有提交调试用的 `console.log` / `print` 等临时代码
- [ ] 没有意外提交 `.env`、`node_modules`、`*.db`、`uploads/*` 等不应入库的文件

### 4.2 创建 PR

在 GitHub / GitLab 等平台创建 Pull Request：

1. 推送分支后，访问仓库页面，点击 **"Compare & pull request"**
2. **标题**：简短概括本次变更（≤70 字）
3. **描述**：按以下模板填写

```markdown
## 概要
- 简要说明本次 PR 做了什么（2-3 句话）

## 变更内容
- [ ] 具体改动项 1
- [ ] 具体改动项 2
- [ ] ...

## 测试计划
- [ ] 正向流程验证（描述如何测试）
- [ ] 异常场景验证（描述如何测试）
- [ ] 边界条件验证（描述如何测试）

## 截图/录屏（如有 UI 变更）
（粘贴截图或录屏链接）

## 关联
- 关联 PRD：xxx
- 关联 Issue：#xxx
```

### 4.3 代码评审

1. **指定 Reviewer**：至少指定 1 名团队成员进行 Code Review
2. **响应反馈**：对 Reviewer 的评论逐条回复或修改
3. **修改后推送**：在本分支修改代码后直接 `git push`，PR 会自动更新

### 4.4 合并 PR

评审通过后，由 **PR 作者** 或 **项目维护者** 合并：

- **Squash and Merge**（推荐）：将所有提交压缩为一个提交，保持 master 历史整洁
- **Merge Commit**：保留完整分支历史，适合多人协作的大型功能

合并后，**删除远程分支**（GitHub 上可以一键删除）。

### 4.5 清理本地分支

```bash
git checkout master
git pull origin master
git branch -d feature/your-feature-name
```

## 5. 常见问题处理

### 冲突解决

合并 master 时出现冲突：

```bash
git merge master
# 出现 CONFLICT 提示
```

1. 打开冲突文件，找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记
2. 手动选择保留哪些代码，删除冲突标记
3. 保存文件后：

```bash
git add <已解决冲突的文件>
git commit -m "merge: 合并 master 并解决冲突"
```

### 撤销提交

```bash
# 撤销最近一次提交，保留修改的文件（推荐）
git reset --soft HEAD~1

# 彻底撤销最近一次提交及修改
git reset --hard HEAD~1
```

### 临时保存工作

切换分支但不方便提交时：

```bash
git stash            # 暂存当前修改
git stash list       # 查看暂存列表
git stash pop        # 恢复最近一次暂存
```

## 6. 分支保护规则（建议）

项目维护者可在仓库设置中配置：

| 规则 | 说明 |
|------|------|
| 禁止直接推送到 master | 所有变更必须通过 PR |
| PR 需至少 1 人 Approve | 强制 Code Review |
| PR 前必须通过 CI | 构建/测试通过后方可合并 |
| 合并前必须同步 master | 避免合并冲突 |
| Squash Merge 作为默认 | 保持 master 历史干净 |

## 7. 快速指令速查

```bash
# 开始新功能
git checkout master && git pull origin master
git checkout -b feature/my-feature

# 日常提交
git add .
git commit -m "feat: xxxx"
git push

# 同步 master
git checkout master && git pull origin master
git checkout feature/my-feature && git merge master

# 清理已完成分支
git checkout master && git pull origin master
git branch -d feature/my-feature
```
